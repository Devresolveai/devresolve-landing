Security

Last updated: April 22, 2026

Security is foundational to DevResolve. Customers trust us with their documentation and developer conversations. Here's exactly how we protect that trust.

TLS 1.2+ everywhere SSH key-only access Non-root deployment fail2ban UFW firewall Tenant isolation

Infrastructure

  • All traffic is encrypted with TLS 1.2+ via Caddy with automatic Let's Encrypt certificates
  • Production servers use SSH key-only authentication — password login is disabled
  • Non-root deployment user with Docker access only — no sudo
  • UFW firewall allows only ports 80, 443, and SSH
  • fail2ban blocks brute-force SSH attempts automatically
  • Unattended security upgrades enabled on all production hosts

Data Isolation

  • All database queries are scoped to org_id — cross-tenant data access is architecturally impossible
  • Vector embeddings are namespaced per org in Pinecone — no cross-org search
  • API keys are stored hashed — we cannot retrieve your plaintext key
  • Webhook secrets use HMAC-SHA256 — we verify every inbound webhook

Knowledge Base Security

  • Raw document content is never stored permanently — only vector embeddings
  • LLM API calls (Anthropic/OpenAI) include only the relevant retrieved snippets and the user question — no bulk document transfer
  • Confluence/Notion API tokens are stored encrypted in your org record

Sovereign Plan

For customers with strict data residency or compliance requirements, the Sovereign plan runs DevResolve entirely within your own VPC or on-premise infrastructure. Zero data passes through DevResolve servers. You supply your own LLM API keys (BYOK). DevResolve has no visibility into your data, queries, or responses.

Vulnerability Disclosure

If you discover a security vulnerability in DevResolve, please email devresolve.ai@outlook.com with details. We will respond within 48 hours and keep you updated as we address the issue. We do not currently offer a bug bounty programme but treat responsible disclosure seriously.
